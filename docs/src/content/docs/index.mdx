## Why grx

<CardGrid>
  <Card title="Fast by construction" icon="rocket">
    On the [bundled benchmarks](/benchmarks/), grx executes a simple query
    in **~2.7 µs** with **39 allocations** — roughly **25× faster** and
    **22× fewer allocations** than `graphql-go`.
  </Card>
  <Card title="Zero runtime dependencies" icon="seti:go">
    A single Go module, `go 1.22`, no third-party imports. The whole runtime
    is auditable, vendorable, and easy to upgrade.
  </Card>
  <Card title="Code-first schemas" icon="document">
    Define your schema as plain Go structs and methods. Reflection runs once
    at startup; the per-request hot path stays allocation-aware.
  </Card>
  <Card title="Built-in subscriptions" icon="forward-slash">
    Ship realtime out of the box with the `graphql-transport-ws`
    subprotocol over WebSocket and GraphQL over Server-Sent Events.
  </Card>
  <Card title="Pluggable transports" icon="puzzle">
    Anything that speaks GraphQL — HTTP+JSON, WebSocket, SSE, or your own —
    is just a `core.Transport` implementation registered on the server.
  </Card>
  <Card title="Pub/Sub built in" icon="random">
    A typed `pubsub.PubSub` primitive with an in-process default and an
    optional Redis backend lets mutations fan events to subscriptions
    across goroutines or replicas.
  </Card>
  <Card title="Plugin lifecycle" icon="setting">
    Hook into request, parse, validate, execute, field-resolve, and response
    phases. Compose logging, tracing, and auth without touching the core.
  </Card>
</CardGrid>

## Project status

grx implements a useful subset of the
[GraphQL October 2021 specification](https://spec.graphql.org/October2021/) with
production-grade transports for queries, mutations, and subscriptions. It is
under active development; the
[implementation checklist](https://github.com/patrickkabwe/grx#feature-parity-checklist)
in the README is the source of truth for what is and isn't supported today.
